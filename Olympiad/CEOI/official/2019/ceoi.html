1192A - Building Skyscrapers

Let's look at the problem in reverse: 𝑛n times, we want to find an arbitrary non-8-articulation 4-reachable from infinity and remove it. This doesn't work if we can reach a situation where no such cell exists, so we need to prove that each 8-connected figure contains at least one non-articulation 4-reachable from infinity.

Proof: If there is a cell with 1 neighbour that is 4-reachable, we are done. Therefore, let's assume all 4-reachable (we always take this to mean "from infinity") cells have at least 2 neighbours.

For any pair of 4-adjacent cells (𝑐1,𝑐2)(c1,c2) such that 𝑐1c1 is full and 𝑐2c2 empty, the region of (𝑐1,𝑐2)(c1,c2) is the set of empty cells which are 8-adjacent to 𝑐1c1 and 4-reachable from 𝑐2c2 by only moving between these cells. The regions for fixed 𝑐1c1 can be found simply by looking at all 8-neighbours of 𝑐1c1 in cyclic order and removing full cells from this cycle; connected segments are regions.

We can now find an 8-cycle of full 4-reachable cells and a corresponding 4-path of empty cells. First, let's pick a pair (𝑐𝑓,𝑐𝑒)=(𝑐𝑓𝑖,𝑐𝑒𝑖)(cf,ce)=(cfi,cei) = (a 4-reachable full cell, an empty cell 4-reachable from infinity and 4-adjacent to it). Then, let's build a sequence of such pairs by repeating the following:

  1. Find the region 𝑅R of (𝑐𝑓,𝑐𝑒)(cf,ce).
  2. Find 2 full cells that border this region and are 8-adjacent to 𝑐𝑓cf (i.e. full cells that were removed to create this region); two such distinct cells must exist since 𝑐𝑓cf has at least 2 full neighbours.
  3. Choose one of these cells (let's denote it by 𝑐𝑓𝑛cfn) that was not chosen in the sequence immediately before 𝑐𝑓cf and the 4-neighbour of 𝑐𝑓𝑛cfn (let's denote it by 𝑐𝑒𝑛cen) from region 𝑅R. Note that this relation is bidirectional, i.e. if we start from (𝑐𝑓𝑛,𝑐𝑒𝑛)(cfn,cen), then 𝑐𝑒ce is in the region of (𝑐𝑓𝑛,𝑐𝑒𝑛)(cfn,cen) and 𝑐𝑓cf borders this region.
  4. Change (𝑐𝑓,𝑐𝑒)(cf,ce) to (𝑐𝑓𝑛,𝑐𝑒𝑛)(cfn,cen).

This is repeated until we reach 𝑐𝑓𝑛=𝑐𝑓𝑖cfn=cfi (initial 𝑐𝑓cf). The process must terminate since the number of full cells is finite. At the end, the full cells from this sequence form a simple 8-connected cycle 𝐶C and the empty cells form a 4-connected path 𝑃P. Each of the full cells from this cycle is therefore 4-reachable from infinity.

Now, let's compute the minimum 𝑟r, minimum 𝑐c, maximum 𝑟r and maximum 𝑐c coordinates of empty cells from cycle 𝐶C. Since the cycle contains at least 2 cells, there must be a cell 𝑐0c0 which is distinct from 𝑐𝑓𝑖cfi and at least one of its coordinates (𝑟,𝑐)(r,c) is one of these four extremum coordinates. We need to show that this cell is not an articulation.

It is sufficient to show that 𝑐0c0 has 3 empty 8-neighbours on one side, i.e. the 3x3 square around it looks as follows (center: 𝑐0c0, top center: empty cell 4-reachable from infinity).

If it has this form, the top 3 cells are in the same region and if we find the full cells bordering this region (among cells marked *), then these cells are on cycle 𝐶C as well, since 𝑐0c0 is not 𝑐𝑓𝑖cfi. Each cell marked * is 8-adjacent to one of these full cells, so emptying the central cell does not break connectivity.

W.l.o.g. let the top 3 cells in the figure above be outside the (min r, max r)×(min c, max c)(min r, max r)×(min c, max c) rectangle. Let's prove it by contradiction – assume that there is a full cell among those three; this cell cannot be part of cycle 𝐶C. Since path 𝑃P consists of cells adjacent to those from cycle 𝐶C, it cannot contain any cells above this figure, so the top middle cell cannot be part of path 𝑃P either (it is not 4-adjacent to 𝑐𝑓𝑖cfi and it is either full or has at most 1 empty neighbour in the top row). Path 𝑃P and cell 𝑐𝑓𝑖cfi form a 4-cycle, which splits the grid into two 8-components, one of which contains all cells from 𝐶C other than 𝑐𝑓𝑖cfi. Since everything above our figure does not contain any cells from 𝑃+𝑐𝑓𝑖P+cfi, it must be part of the latter component, and since it is 4-connected to the top middle cell, which cannot be part of 𝑃+𝑐𝑓𝑖P+cfi, the top middle cell must also be part of this component. But then, the middle cell is 4-adjacent to the second component and it is part of the first component, which is a contradiction. QED.

Hopefully, you've had enough math by now. (If not, you can try to prove the same claim for 4-connected full cells and 8-adjacency to empty cells 8-reachable from infinity.) We can finally get back to the programming problem!

Subtasks 4, 6 and 7

Checking reachability from infinity is easy. We can store the 4-connected components of empty cells, remember which component corresponds to the outside and whenever we make a new empty cell, we can update this information with the simple "merge smaller to larger" strategy. We just have to choose which empty cells to consider, since the naive estimate is (𝑛2)O(n2) empty cells; however, empty cells which have no full 8-neighbours in the original figure are not interesting, so we are left with (𝑛)O(n) interesting cells and the total time complexity of calculating connected components is (𝑛log𝑛)O(nlog⁡n).

When is a full cell 𝑐c an articulation? Let's look at all its regions that aren't isolated empty corners (since their neighbours on the border of 𝑐c are 8-connected, so the following argument doesn't apply). For each pair of different regions 𝑟1r1 and 𝑟2r2, if there is no 4-path of empty cells connecting them, then for each pair of full non-adjacent 8-neighbours of 𝑐c (𝑐1c1 and 𝑐2c2) such that 𝑐1,𝑟1,𝑐2,𝑟2c1,r1,c2,r2 are in this order around 𝑐c, there is an 8-path of full cells connecting 𝑐1c1 and 𝑐2c2 that does not contain 𝑐c. Vice versa, if there is a path connecting two such cells 𝑐1c1 and 𝑐2c2 that does not contain 𝑐c, then there is no 4-path connecting 𝑟1r1 and 𝑟2r2. Since for each pair of non-corner regions 𝑟1r1 and 𝑟2r2, we can find 2 cells 𝑐1c1 and 𝑐2c2, we get an equivalent definition of a non-articulation: all non-corner regions belong to different 4-connected components of empty cells.

This simplifies things. We're already updating connected components, so the information about articulation-ness of cells is local – if we know the empty 4-neighbours and connected components of a full cell, we can compute whether it is an articulation in (1)O(1) time. This information only needs recomputing if empty neighbours appear or are moved to a different component.

Now, we only need to realise that we already have to look at all empty cells that appear or are moved to a different component. It's done in the "merge smaller to larger" union-find for empty cells – when merging component 𝐾𝑠Ks into component 𝐾𝑙Kl, we have to look at all cells from 𝐾𝑠Ks to update which component they belong to, and we only create an empty cell (𝑛)O(n) times along with that. For each cell at which we look in the union-find, we can just check all its full 8-neighbours, and for each of these neighbours, we can update the information abouts its neighbours, decide whether it is an articulation and if it is reachable from the outer component, all in (1)O(1) time.

We are left with updates saying "this cell can be removed" / "this cell cannot be removed". With them, it's easy to keep the set of removable cells in amortised (log𝑛)O(log⁡n) and choose the cell to remove in each step in (log𝑛)O(log⁡n). The subtask type is irrelevant, we could just as well support queries "remove this cell or report that it's not allowed".

Obviously, the answer is "NO" if and only if the final configuration isn't 8-connected.

Total time complexity: (𝑛log𝑛)O(nlog⁡n), memory complexity (𝑛)O(n).

In subtask 6, we can afford to store the graph as a 2D array, with complexity (𝑛log𝑛+𝑟𝑚𝑎𝑥𝑐𝑚𝑎𝑥)O(nlog⁡n+rmaxcmax). Also, we are checking if a cell is an articulation in (𝑑)O(d), where 𝑑=8d=8 is the number of neighbours of a cell, but less efficient implementations like finding regions and checking all pairs of regions in (𝑑2)O(d2) become possible.

In subtask 4, we only need to determine if it's possible to remove a skyscraper in (1)O(1). Whenever we're removing a skyscraper, we can simply find the last removable one, then update the connected components, either by merging existing ones or running BFS on the interesting cells. This works in (𝑛2)O(n2).

Subtasks 3 and 5

When we are allowed to choose the output sequence, a much simpler approach becomes possible:

  • First, build an arbitrary skyscraper.
  • Store the set of skyscrapers which haven't been built, but are 8-adjacent to those which have been built.
  • While we need to build a skyscraper, pick the rightmost one from this set (with the maximum 𝑐𝑖ci); if there is more than one such skyscraper, pick the one with the maximum 𝑟𝑖ri. Then, update the set.

This takes (𝑛log𝑛)O(nlog⁡n) time when the skyscrapers are stored efficiently (e.g. in an STL set<>), or (𝑛2)O(n2) by brute force.

The set of built skyscrapers is 8-connected, but it isn't immediately clear why we can't fail by closing an 8-cycle and then building a skyscraper inside it. However, it makes sense – informally, before building the last skyscraper from this cycle, we can already build everything on its inner border, and if the final configuration is 8-connected and there is something inside the cycle we still need to build, there will be a skyscraper we haven't built yet on this border with greater 𝑐𝑖ci (or equal 𝑐𝑖ci and greater 𝑟𝑖ri). We'll build everything inside a cycle before we finish the cycle itself.

Subtask 2

We can use the approach for 𝑡=1t=1, or we can guess that the hypothesis about existence of a removable skyscraper is true. Then, we can choose a skyscraper to remove by checking which ones can be removed (by removing and running BFS on the remaining graphs of empty and full cells). If we do this 𝑛n times, the time complexity is (𝑛3)O(n3).

Subtask 1

We can try all (𝑛!)O(n!) possible orders and check if they are valid.

1192B - Dynamic Diameter

Subtask 1

In the first subtask the limits are small enough to do literally what is written in the problem statement – after each update we run DFS from each vertex to compute all pairs of distances and report the largest one.

Complexity (𝑞𝑛2)O(qn2).

Subtask 2

In the second subtask, we need to compute the diameter in (𝑛)O(n). There are two ways:

  • The first one is a somewhat standard trick. Use DFS from vertex 11 to find the most distant vertex, let's call it 𝑢u. The diameter is then the distance to the most distant vertex from 𝑢u, which can be again found using DFS.
  • The second approach is using a dynamic programming on a tree, which will be easier to generalise. Root the tree in vertex 11. For each vertex 𝑣v, compute 𝐷𝑣Dv – the maximum distance to leaf in a subtree of 𝑣v. This can be done in (𝑛)O(n) recursively. Then the answer is just maximum (through all vertices 𝑢u) of 𝐷𝑢Du and max𝑣1≠𝑣2 are children of 𝑢(𝐷𝑣1+𝐷𝑣2+𝑤𝑢,𝑣1+𝑤𝑢,𝑣2)maxv1≠v2 are children of u(Dv1+Dv2+wu,v1+wu,v2)

Complexity (𝑞𝑛)O(qn).

Subtask 3

When the graph is a star, it means that the diameter consists only of two edges, both of which are incident to vertex 11. We can thus simply maintain the weight of each edge in a multiset and output the sum of the two largest values as an answer. Be careful about 𝑛=2n=2.

Complexity ((𝑞+𝑛)log𝑛)O((q+n)log⁡n)

Subtask 4

In the fourth subtask, we modify the DP from the second subtask slightly by introducing a new value 𝑆𝑢Su to be the diameter of the subtree rooted at 𝑢u. We can see that

𝑆𝑢=max(𝐷𝑢,max𝑣 child of 𝑢𝑆𝑣,max𝑣1≠𝑣2 children of 𝑢𝐷𝑣1+𝐷𝑣2+𝑤𝑢,𝑣1+𝑤𝑢,𝑣2)Su=max(Du,maxv child of uSv,maxv1≠v2 children of uDv1+Dv2+wu,v1+wu,v2)

and the diameter equals 𝑆1S1.

Upon changing the weight of the edge {𝑢,𝑣}{u,v}, only the values 𝑆𝑥,𝐷𝑥Sx,Dx where 𝑥x is an ancestor of 𝑢u need to be updated. As each update can be processed in constant time (the number of immediate children of each vertex is at most 22) and as the tree is balanced, it's depth is (log𝑛)O(log⁡n).

Overall this needs (𝑛+𝑞log𝑛)O(n+qlog⁡n) time.

Subtask 5

Assume that the diameter always goes through vertex 𝑢u. This means that the diameter is found by picking two children of vertex 𝑢u – 𝑣1v1 and 𝑣2v2 – such that 𝐷𝑣1+𝐷𝑣2+𝑤𝑢,𝑣1+𝑤𝑢,𝑣2Dv1+Dv2+wu,v1+wu,v2 is maximised.

To calculate each value of 𝐷𝑣𝑖Dvi, we can use a segment tree that maintains the distances of each vertex to root. When the vertices are ordered using DFS, an update becomes an addition on a range, so any segment tree with lazy propagation is sufficient.

To calculate the maximum of the sum above, we use the approach outlined in subtask 3 – we maintain a multiset of 𝐷𝑣𝑖+𝑤𝑢,𝑣𝑖Dvi+wu,vi and pick the largest two values.

In subtask 5, we can simply assign 𝑢=1u=1. This needs (𝑞log𝑛)O(qlog⁡n) time.

Subtask 6

In subtask 6 we cannot pick any single vertex through which all the diameters traverse. Nevertheless, we can still pick a vertex 𝑢u and find the longest path that goes through vertex 𝑢u to get a lower bound on the answer. Then, we remove 𝑢u from the tree, leaving us with some number of smaller trees. This approach can be recursively applied to them.

Performed naively, this leads to a quadratic performance – for example consider a path on 𝑛n vertices, where we always select a leaf as the root. We need to limit the depth of this recursion in some way.

We do this by always picking the centroid as a root. Recall that centroid is a vertex that when removed splits a tree into a number of subtrees each having at most 𝑛/2n/2 vertices where 𝑛n is the size of the original tree. Every tree has 11 or 22 centroids that can be found in linear time using a simple DP.

If we always pick a centroid, then in (log𝑛)O(log⁡n) iterations we end up with a couple of trees of size 11, for which the answer is 00, so this can be done in (𝑛log𝑛)O(nlog⁡n).

Now we simply maintain the segment tree for each centroid. As each edge is in (log𝑛)O(log⁡n) trees, each update can be performed in (log2𝑛)O(log2⁡n). The overall complexity is ((𝑛+𝑞)log2𝑛)O((n+q)log2⁡n).

Subtask 6 – alternative approach

Let's root the tree. Recall the trick from subtask 2: To compute the diameter, we should find the deepest (most distant from root) vertex of the tree – let's call it 𝑢u – and then find the maximum distance from 𝑢u. Since we cannot expect anything about 𝑢u, this problem can also be formulated as just finding the farthest vertex from a given vertex.

Instead of centroid decomposition, we can use heavy-light decomposition. For each vertex 𝑣v, there is at most one son 𝑠s connected to 𝑣v by a heavy edge. Let 𝑚𝑣mv denote the maximum of depths of all descendants of 𝑣v which are not descendants of 𝑠s, inclusive. On each path of the HLD, we should build a segment tree which stores the values 𝑚𝑣−2𝑑𝑣mv−2dv, where 𝑑𝑣dv is the depth of 𝑣v. Also, let's measure these depths from the top of the path containing 𝑣v instead of from the root.

When looking for the maximum distance from vertex 𝑢u, let's move to the root and look at all ancestors of 𝑢u (including 𝑢u itself). When we take the son 𝑠s from which we reached 𝑣v and compute the maximum 𝑚m of depths of descendants of 𝑣v which aren't descendants of 𝑠s (if 𝑣=𝑢v=u, there are no such vertices because 𝑠s doesn't exist), then the maximum of lengths of all paths from 𝑢u which have 𝑣v as the LCA is 𝑚+𝑑𝑢−2𝑑𝑣m+du−2dv, where the depths are again measured from the top of the path containing 𝑣v. The maximum distance is the maximum of all these lengths and it's easy to extend the process that computes it to find the vertex with this distance.

Of course, we can't afford to do that by brute force, but HLD helps. If the last edge we traversed in order to reach an ancestor 𝑣v was a heavy edge, then 𝑚=𝑚𝑣m=mv. This isn't the case only when we just reached a new heavy path; there are (log𝑛)O(log⁡n) heavy paths on any path to the root and therefore just (log𝑛)O(log⁡n) such vertices. When we're moving from 𝑢u to the root and reach a path 𝑝p in a vertex 𝑣′v′, we need to compute 𝑚+𝑑𝑢−2𝑑𝑣′m+du−2dv′ for this vertex. Then, we know that we must visit all vertices 𝑣v above 𝑣′v′ on this path, so the maximum of 𝑚𝑣+𝑑𝑢−2𝑑𝑣mv+du−2dv for all these vertices is a prefix maximum from the segment tree of path 𝑝p, plus 𝑑𝑢du.

How to compute 𝑚m? With preorder numbering of the vertices, each subtree becomes a segment, so we just need a segment tree that stores depths from the root and 𝑚m is the maximum from two segments minus the depth of the top of the current path. This segment tree lets us find depths of individual vertices as well.

Now, we just need to handle updates. When we add 𝑎a to the weight of an edge from 𝑢u to 𝑣v (𝑣v is a son of 𝑢u), we add 𝑎a to the depths from the root of all vertices in the subtree of 𝑣v. The values of 𝑚m can only change for 𝑢u and the ancestors of 𝑢u which are reached by a light edge. There are only (log𝑛)O(log⁡n) of them, so we can afford to recompute 𝑚m for each of them. Recall that we're storing 𝑚−2𝑑m−2d in the tree, so we need to also subtract 2𝑎2a from all vertices on the path containing the updated edge, located below that edge (𝑑d changes only for these vertices – this is why we aren't using depths from the root). Therefore, the segment trees built over HLD paths need to support operations "update value", "add to segment" and "get maximum from segment".

The time complexity of this approach is (𝑛+𝑞log2𝑛)O(n+qlog2⁡n), since we're handling (log𝑛)O(log⁡n) paths per query and doing (1)O(1) segment tree operations per path.

1192C - Cubeword

In all subtasks we solve the problem independently for each possible length of a word and sum the answers.

To handle palindromes and words that are reverse of each other, we first add the reverse of each word to the dictionary and then remove duplicates. Then, we will fix a direction for each edge and we will ask that the word written along the edge in that direction has to be in the new wordlist. (This is clearly equivalent to the original problem, and it allows us to choose the directions of edges in any way that suits our approach.)

For each pair of letters 𝑎a, 𝑏b precompute the number of words that begin with 𝑎a and end in 𝑏b.

Subtask 1

We brute-force all combinations of letters in the vertices of the cube and use the precomputed values to find how many cubewords are there with these letters fixed. Complexity (𝑛+|Σ|8)O(n+|Σ|8).

Subtask 2

Label the vertices 11 through 88 with vertices 11 through 44 lying around the bottom face and vertex 𝑖+4i+4 directly above vertex i. We will assign letters to vertices in the increasing order of their ids. Start by assigning letters to vertices 11 through 44 and count the number of possibilities of assigning letters to the edges on the bottom face. We then try all possibilities of assigning a letter to vertex 55, and account for the edge {1,5}{1,5}. Note that all edges incident to the vertex 11 are already processed, hence we may now drop the value of 11 from our state. More generally, after processing vertex 𝑗j we can forget the letter in vertex 𝑗−4j−4.

This forms a dynamic programming solution with 8⋅|Σ|48⋅|Σ|4 states, where each state has |Σ||Σ| transitions, each of which can be performed in constant time.

Complexity (𝑛+|Σ|5)O(n+|Σ|5).

Subtask 3

When viewed as a graph, the cube is bipartite with both partitions of size 44. Consider all possibilities of assigning the letters to one partition. We want to process each of the remaining vertices in (1)O(1).

Precompute 𝐷[𝑎][𝑏][𝑐]D[a][b][c] as the number of triples of words such that they all begin with the same letter and the other end with 𝑎a,𝑏b,𝑐c, respectively. This precomputation can be done naively in (|Σ|4)O(|Σ|4).

Coming back to our problem, let the letters on one partition be 𝑎a,𝑏b,𝑐c,𝑑d respectively. Observe that the answer is 𝐷[𝑎][𝑏][𝑐]⋅𝐷[𝑎][𝑏][𝑑]⋅𝐷[𝑎][𝑐][𝑑]⋅𝐷[𝑏][𝑐][𝑑]D[a][b][c]⋅D[a][b][d]⋅D[a][c][d]⋅D[b][c][d].

Total complexity is (𝑛+|Σ|4)O(n+|Σ|4).

Fun fact: Geometrically, this solution can be viewed as follows. A cube can be cut into five tetrahedrons: one that's completely inside and four such that each of them contains one corner of the cube (and thus three edges). In this solution you brute force the letters in the corners of the inner tetrahedron, and once you fix those, each of the remaining tetrahedrons is independent of the other three.

Subtask 4

To reduce the complexity even further, note that a cube can be rotated and flipped in 4848 different ways: there are 66 possibilities for the bottom face, after the bottom face is fixed, there are 44 options for the front face, and after that there are two possibilities on the left face. In exactly half of the rotations, the vertex partitions swap places (for instance when rotating the cube by 90 degrees in any principal axis) and in the second half they stay in position, possibly permuting the vertices (for example rotation the cube by 180 degrees, or rotating the cube by 90 degrees and then mirroring it).

Assume for a moment that the letters 𝑎a, 𝑏b, 𝑐c, 𝑑d on the vertices of the first partition are guaranteed to be distinct. Then there is a way of rotating and flipping the cube in such a way that 𝑎<𝑏<𝑐<𝑑a<b<c<d. Hence we can calculate the answer only for this case and multiply it by 2424.

However, the letters 𝑎a, 𝑏b, 𝑐c, 𝑑d are not necessarily distinct. To mitigate this, we simply consider all possibilities of the form 𝑎≤𝑏≤𝑐≤𝑑a≤b≤c≤d, and multiply the answer of each of them by the number of permutations of (𝑎,𝑏,𝑐,𝑑)(a,b,c,d). This reduces this step of the solution to (|Σ|4/24)O(|Σ|4/24).

Precomputing the array 𝐷D can also be optimised – we again consider only cases 𝑎≤𝑏≤𝑐a≤b≤c and use the fact that 𝐷[𝑎][𝑏][𝑐]=𝐷[𝑎][𝑐][𝑏]=…=𝐷[𝑐][𝑏][𝑎]D[a][b][c]=D[a][c][b]=…=D[c][b][a]. The complexity of this step is thus reduced to (|Σ|4/6)O(|Σ|4/6).

To summarise, we are able to reduce the complexity 66-fold by considering symmetries.

If you have the main idea (that you need to write one function to locally decide removability and how to do that), which is the hard part, everything else is quite simple. My code is written for clarity and some efficiency rather than code golf and it has "just" 300 lines.

      1. You need to parse the input into a graph with ≤9𝑁≤9N vertices and do a BFS/DFS to find components. There's no way around this, but it's all very easy.
      2. You need to write the "update" function. All decisions about articulations and reachability are contained here. It uses graph information: components, degrees, which cells are empty and the graph itself.
      3. When you remove a cell, you need to merge components and update the degrees. This is classic union-find. In the process, with the slower version of union-find where you don't maintain a tree of merges, but just merge smaller components to larger ones, you're looking at all vertices whose component changed. Look at their neighbours and run update() for them. Also run update() for neighbours of the cell you just emptied.
